### コンテキスト候補

- **コア**: 勤怠管理、申請承認、休暇管理
- **支援**: 月次処理
- **汎用**: 通知

### 主要フロー

1. 打刻→勤務時間計算→残業チェック
2. 残業申請→承認→勤怠記録反映
3. 休暇申請→承認→有給消化→勤怠記録反映
4. 月次仮締め→確認→本締め→CSVエクスポート

## 注意点

- 申請承認コンテキストが勤怠管理と休暇管理の両方に影響する**ハブ的存在**。ES図ではコンテキスト間の連携（Saga）に注意
- フレックスタイム制の残業計算は**月単位清算**で、日次の勤怠記録とは別に月次集計が必要
- 夜勤の勤務日判定（開始日=勤務日）と深夜割増（22:00-5:00）の計算ロジックは、勤怠記録集約の中核ドメインロジック
- 「本締め後修正不可」はイミュータブルの設計判断に直結する
- Session 3 で「カレンダー同期は双方向」との初回回答を「一方向プッシュ」に訂正した（会話記録で確認可能）
